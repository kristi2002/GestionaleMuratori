## Decisions locked (grounded in code review)

- **Reservation model = Additive (site-optional).** *Decision made after reviewing the code.*
  Interventions keep reserving from the **main warehouse (location 1)** by default;
  `qty_in_stock` continues to mean "location-1 balance", so **every existing invariant and all
  174 tests stay valid**. Per-site balances and warehouseâ†”site transfers are added *underneath*
  as a new capability; `InterventionService::create/complete` gain an **optional** `locationId`
  defaulting to `1`. This is a strict superset â€” a future "site-first default" (Phase 4+) becomes
  a one-line default change, but we don't rewrite create/complete or existing stock tests inside
  the same PR that already carries the risky location-ledger schema change + a bug fix. The
  draft's "materials must be transferred to site before any intervention can consume" behavior is
  documented in the roadmap as a deferred toggle, not built now.
- **Ledger stays the single source of truth.** `stock_balances(item_id, location_id)` is a cache
  (the per-location analogue of `qty_in_stock`), never written without a ledger row â€” mirroring
  the existing `WarehouseItemModel::recomputeStock` discipline.

## Confirmed findings driving the build

- **Migration runner** (`database/migrate.php`): tracks applied files in a `migrations` table,
  applies `migrations/*.sql` in filename-sort order, splits on `;`+newline, strips **only
  full-line `--` comments**, and has **no per-file transaction** (DDL auto-commits in MySQL).
  â†’ New files must be `003_â€¦`â€“`009_â€¦`, every statement terminated by `;`-newline, comments on
  their **own** lines, and each file safe to re-run conceptually / independently valid.
- **Ledger sign convention** lives in one place â€” `WarehouseItemModel::recomputeStock`
  (`src/Models/WarehouseItemModel.php`): `SUM(CASE WHEN type IN('in','release') THEN qty
  WHEN type='reserve' THEN -qty WHEN type='adjustment' THEN qty ELSE 0 END)`. `out` is weight-0.
- **Confirmed latent bug** â€” `InterventionService::complete()` (`src/Services/InterventionService.php:209â€“236`):
  the loop iterates **all** materials from `forInterventionForUpdate` (unfiltered) and emits a
  `release` of `qty_planned âˆ’ qty_used` on **every** row **without checking `is_reserved`**.
  A never-reserved row (no offsetting `reserve` row) therefore adds phantom stock. It's reachable
  **today**: `database/seed.php` inserts `is_reserved=0` materials on a seeded `in_progress`
  intervention. The correct pattern already exists in `transition('cancelled')` at line 128
  (`reservedForUpdate` + `markReleased`).
- `projects` **already has `lat`/`lng`** (used later by weather). `warehouse_items` has **no
  cost column**. `users.role` is `admin|worker|client` only. Models are stateless `final class`,
  namespace `App\Models`, use `App\Support\Database::pdo()`, positional `?` for lookups / named
  `:x` for writes, `â€¦ForUpdate()` for locking reads. JSON contract via `App\Support\Response`
  (`ok`/`fail`). Lang via `App\Support\Lang::get/label` + `lang/it.php` (no `subcontractor` block yet).
